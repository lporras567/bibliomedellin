01

Infraestructura y Seguridad

Roles, bases de datos y principio de mínimo privilegio

Fundamentación Teórica

🔐
Roles en PostgreSQL

Un rol es una entidad que puede actuar como usuario (si tiene LOGIN) o como grupo. La gestión de roles permite separar responsabilidades entre proyectos.

🛡️
Principio de Mínimo Privilegio

El superusuario postgres nunca debe usarse en desarrollo. Se crea un administrador específico del proyecto con privilegios limitados y controlados.

👑
OWNER de la BD

El propietario (OWNER) tiene control total sobre su base de datos, garantizando que solo procesos autorizados realicen cambios estructurales.

A Acceso inicial como superusuario
bash 
psql -U postgres
B Crear el Rol Administrador
sql 
-- Creamos el rol con permiso de inicio de sesión y contraseña cifrada
CREATE ROLE admin_biblioteca WITH
    LOGIN
    PASSWORD 'Medellin2026*'
    CREATEDB
    CREATEROLE;

COMMENT ON ROLE admin_biblioteca IS
    'Administrador principal de la DB Biblioteca Municipal';
C Crear la Base de Datos
sql 
-- Creamos la BD asignando el propietario desde la creación
CREATE DATABASE biblioteca_medellin
    OWNER    admin_biblioteca
    ENCODING 'UTF8';

-- Conectarse a la nueva base de datos
\c biblioteca_medellin

-- Otorgar privilegios totales sobre el esquema público
GRANT ALL PRIVILEGES ON SCHEMA public TO admin_biblioteca;
D Validación: Prueba de Conexión
bash 
psql -U admin_biblioteca -d biblioteca_medellin
-- Contraseña: Medellin2026*

💡 Tips para la Clase

CREATEDB vs SUPERUSER: Un SUPERUSER puede borrarlo todo; un usuario con CREATEDB solo gestiona las bases de datos que él mismo cree.
pg_hba.conf: Si aparece "Peer authentication failed", PostgreSQL controla accesos según este archivo de configuración.
Naming Convention: Siempre usar nombres en minúsculas con guiones bajos (_) para evitar problemas con comillas dobles en PostgreSQL.
02

Modelado Físico

Tablas, tipos ENUM, integridad referencial e índices

Fundamentación Teórica

🔗
Integridad Referencial

PRIMARY KEY identifica registros únicos. FOREIGN KEY conecta tablas garantizando que no existan registros huérfanos.

↔️
Relaciones N:M

Un libro puede tener varios autores y viceversa. Se resuelve con una tabla intermedia (libro_autor) con PK compuesta.

⚡
Índices

Evitan que PostgreSQL lea toda la tabla (Seq Scan). Aceleran búsquedas en columnas de cláusulas WHERE y JOIN frecuentes.

A Conectarse como admin_biblioteca
bash 
\c biblioteca_medellin admin_biblioteca
B Crear Tablas Principales
sql 
-- 1. Tabla de Autores
CREATE TABLE autores (
    autor_id   SERIAL PRIMARY KEY,
    nombre     VARCHAR(100) NOT NULL,
    nacionalidad VARCHAR(50)
);

-- 2. Tabla de Libros (ISBN como PK natural)
CREATE TABLE libros (
    isbn              VARCHAR(20) PRIMARY KEY,
    titulo            VARCHAR(200) NOT NULL,
    anio_publicacion  INTEGER,
    genero            VARCHAR(50)
);

-- 3. Tabla Intermedia: Libro_Autor (Relación N:M)
CREATE TABLE libro_autor (
    isbn     VARCHAR(20) REFERENCES libros(isbn)   ON DELETE CASCADE,
    autor_id INTEGER     REFERENCES autores(autor_id) ON DELETE CASCADE,
    PRIMARY KEY (isbn, autor_id)
);

-- 4. Tabla de Sedes
CREATE TABLE sedes (
    sede_id    SERIAL PRIMARY KEY,
    nombre_sede VARCHAR(100) NOT NULL,
    direccion   TEXT
);
C Tipos ENUM y Tablas con FK
sql 
-- 5. Tipos enumerados (valores cerrados y controlados)
CREATE TYPE estado_ejemplar AS ENUM
    ('Excelente', 'Bueno', 'Deteriorado', 'En Reparación');

CREATE TYPE estado_prestamo AS ENUM
    ('Activo', 'Devuelto', 'Vencido');

CREATE TYPE tipo_usuario AS ENUM
    ('Estudiante', 'Docente', 'Publico Externo');

-- 6. Tabla de Ejemplares (copia física de un libro en una sede)
CREATE TABLE ejemplares (
    ejemplar_id          SERIAL PRIMARY KEY,
    isbn                 VARCHAR(20) REFERENCES libros(isbn),
    sede_id              INTEGER     REFERENCES sedes(sede_id),
    ubicacion_estanteria VARCHAR(50),
    estado               estado_ejemplar DEFAULT 'Bueno'
);

-- 7. Tabla de Usuarios
CREATE TABLE usuarios (
    usuario_id     SERIAL PRIMARY KEY,
    nombre_completo VARCHAR(150) NOT NULL,
    tipo            tipo_usuario NOT NULL
);

-- 8. Tabla de Préstamos
CREATE TABLE prestamos (
    prestamo_id  SERIAL PRIMARY KEY,
    ejemplar_id  INTEGER REFERENCES ejemplares(ejemplar_id),
    usuario_id   INTEGER REFERENCES usuarios(usuario_id),
    fecha_inicio DATE    DEFAULT CURRENT_DATE,
    fecha_limite DATE    NOT NULL,
    estado       estado_prestamo DEFAULT 'Activo'
);
D Índices de Optimización
sql 
-- Búsqueda rápida por título (consulta más frecuente)
CREATE INDEX idx_libro_titulo
    ON libros(titulo);

-- Búsqueda de usuarios por nombre (reportes)
CREATE INDEX idx_usuario_nombre
    ON usuarios(nombre_completo);

-- Acelerar reportes de morosos y libros vencidos
CREATE INDEX idx_prestamo_estado
    ON prestamos(estado);

📝 Ejercicio de Refuerzo

¿Por qué PK compuesta en libro_autor? La combinación (isbn, autor_id) debe ser única: el mismo autor no puede estar asignado dos veces al mismo libro.
Prueba de FK: Intentar insertar un préstamo con ejemplar_id inexistente y observar cómo el motor rechaza la operación.
SERIAL vs ISBN: Se usa SERIAL para IDs sintéticos y VARCHAR para el ISBN porque ya es un identificador natural universal del libro.

JOINs y Relaciones

Combinar tablas para obtener información cruzada

Tipos de JOIN

INNER JOIN
Solo registros con coincidencia en AMBAS tablas
LEFT JOIN
Todos los de la izquierda + coincidentes de la derecha (NULL si no hay)
FULL OUTER JOIN
Todos los registros de ambas tablas, coincidan o no
A Insertar Datos de Prueba
sql

INSERT INTO sedes (nombre_sede, direccion) VALUES

    ('La Playa',   'Calle 52 # 45-23'),

    ('978-1', 1, 'A-12', 'Excelente'),
    ('978-2', 1, 'B-05', 'Bueno');

-- Usuarios con distintos tipos

B INNER JOIN — Libros con inventario físico
sql 
-- Solo libros que FÍSICAMENTE existen en alguna sede
SELECT
    l.titulo,
    s.nombre_sede,
    e.ubicacion_estanteria,
    e.estado
FROM libros l
INNER JOIN ejemplares e ON l.isbn    = e.isbn
INNER JOIN sedes      s ON e.sede_id = s.sede_id;
-- Resultado: 2 filas (Cien Años y Don Quijote)
-- El código Da Vinci NO aparece porque no tiene ejemplar
C LEFT JOIN — Identificar libros sin inventario
sql 
-- ¿Qué libros tenemos registrados pero sin ejemplar físico?
SELECT
    l.titulo,
    e.ejemplar_id
FROM libros l
LEFT JOIN ejemplares e ON l.isbn = e.isbn
WHERE e.ejemplar_id IS NULL;
-- Resultado: "El código Da Vinci" con ejemplar_id = NULL
D FULL OUTER JOIN — Cruce total usuarios y préstamos
sql 
-- Ver quién ha prestado y quién no
SELECT
    u.nombre_completo,
    p.prestamo_id,
    p.estado
FROM usuarios u
FULL OUTER JOIN prestamos p ON u.usuario_id = p.usuario_id;
🏆 Reto: JOIN de 4 Tablas

Enunciado: "Mostrar el nombre del usuario, el título del libro que prestó y la sede de donde salió el libro"

sql 
-- Solución: navegar por el modelo completo
SELECT
    u.nombre_completo,
    l.titulo,
    s.nombre_sede
FROM usuarios   u
JOIN prestamos  p ON u.usuario_id  = p.usuario_id
JOIN ejemplares e ON p.ejemplar_id = e.ejemplar_id
JOIN libros     l ON e.isbn        = l.isbn
JOIN sedes      s ON e.sede_id     = s.sede_id;

📐 Tips Pedagógicos

Diagramas de Venn: Úsalos para explicar los JOINs visualmente. La intersección = INNER, el círculo completo = LEFT/RIGHT.
Alias de tablas: Enseña a usar l, e, s desde el inicio para que el código sea legible y no ambiguo.
El peligro del Cross Join: Si se olvida la cláusula ON, PostgreSQL produce el producto cartesiano y puede congelar una BD grande.

A Vista: Reporte de Préstamos Críticos
sql 
-- Vista que centraliza JOINs complejos para el reportero
CREATE VIEW vista_prestamos_criticos AS
SELECT
    u.nombre_completo        AS usuario,
    l.titulo                 AS libro,
    s.nombre_sede            AS sede,
    p.fecha_limite,
    (CURRENT_DATE - p.fecha_limite) AS dias_retraso
FROM prestamos  p
JOIN usuarios   u ON p.usuario_id  = u.usuario_id
JOIN ejemplares e ON p.ejemplar_id = e.ejemplar_id
JOIN libros     l ON e.isbn        = l.isbn
JOIN sedes      s ON e.sede_id     = s.sede_id
WHERE p.estado = 'Vencido'
   OR (p.estado = 'Activo' AND p.fecha_limite < CURRENT_DATE);

-- Uso: una sola línea en lugar de 4 JOINs
SELECT * FROM vista_prestamos_criticos;
B Función: Stock disponible por ISBN
sql 
CREATE OR REPLACE FUNCTION fn_stock_disponible(p_isbn VARCHAR)
RETURNS INTEGER AS $$
DECLARE
    v_conteo INTEGER;
BEGIN
    SELECT COUNT(*) INTO v_conteo
    FROM ejemplares
    WHERE isbn = p_isbn
      AND estado IN ('Excelente', 'Bueno');

    RETURN v_conteo;
END;
$$ LANGUAGE plpgsql;

-- Ejemplo de uso en una consulta
SELECT titulo, fn_stock_disponible(isbn) AS unidades_aptas
FROM libros;
C Procedimiento: Procesar Devolución
sql 
CREATE OR REPLACE PROCEDURE sp_procesar_devolucion(
    p_prestamo_id        INTEGER,
    p_nuevo_estado_fisico estado_ejemplar
)
AS $$
BEGIN
    -- 1. Marcar el préstamo como devuelto
    UPDATE prestamos
    SET estado = 'Devuelto'
    WHERE prestamo_id = p_prestamo_id;

    -- 2. Actualizar el estado físico del ejemplar vinculado
    UPDATE ejemplares
    SET estado = p_nuevo_estado_fisico
    WHERE ejemplar_id = (
        SELECT ejemplar_id FROM prestamos
        WHERE prestamo_id = p_prestamo_id
    );

    RAISE NOTICE 'Devolución procesada y estado del ejemplar actualizado.';
END;
$$ LANGUAGE plpgsql;

-- Ejemplo: devolver el préstamo 1, el libro volvió deteriorado
CALL sp_procesar_devolucion(1, 'Deteriorado');

💡 Tips de Instructor

DECLARE: Explica el uso de variables locales dentro del bloque BEGIN...END. Son visibles solo dentro de la función o procedimiento.
Manejo de Errores: Los bloques EXCEPTION capturan errores y evitan fallos silenciosos cuando un ID no existe.
Seguridad: Los procedimientos almacenados previenen SQL Injection porque el usuario ejecuta el SP, no sentencias SQL directas.

Triggers y Automatización

Disparadores que garantizan integridad sin depender del frontend

Anatomía de un Trigger

Evento
INSERT UPDATE DELETE TRUNCATE
→
Momento
BEFORE AFTER
BEFORE = validaciones
AFTER = auditoría/actualizar otras tablas
→
Granularidad
FOR EACH ROW
Actúa sobre cada fila afectada. NEW = datos entrantes, OLD = datos anteriores
→
Función Trigger
RETURNS TRIGGER
Se crea primero la función, luego se asocia al trigger
A Trigger: Gestión automática del estado del ejemplar
sql 
-- Paso 1: Crear la FUNCIÓN que ejecutará el trigger
CREATE OR REPLACE FUNCTION fn_trg_gestionar_ejemplar()
RETURNS TRIGGER AS $$
BEGIN
    -- Caso INSERT: se presta el libro → cambiar estado del ejemplar
    IF (TG_OP = 'INSERT') THEN
        UPDATE ejemplares
        SET estado = 'Deteriorado'
        WHERE ejemplar_id = NEW.ejemplar_id;
        RAISE NOTICE 'Disparador: Estado del ejemplar actualizado por nuevo préstamo.';

    -- Caso DELETE: se cancela el préstamo → liberar el ejemplar
    ELSIF (TG_OP = 'DELETE') THEN
        UPDATE ejemplares
        SET estado = 'Bueno'
        WHERE ejemplar_id = OLD.ejemplar_id;
        RAISE NOTICE 'Disparador: Ejemplar marcado como disponible.';
    END IF;

    RETURN NULL; -- En triggers AFTER el retorno no afecta la operación
END;
$$ LANGUAGE plpgsql;

-- Paso 2: Crear el TRIGGER y asociarlo a la tabla prestamos
CREATE TRIGGER trg_actualizar_ejemplar
AFTER INSERT OR DELETE ON prestamos
FOR EACH ROW
EXECUTE FUNCTION fn_trg_gestionar_ejemplar();
B Trigger de Auditoría: Log de cambios de estado
sql 
-- 1. Tabla de log para registrar cada cambio de estado
CREATE TABLE log_prestamos (
    log_id          SERIAL PRIMARY KEY,
    prestamo_id     INTEGER,
    estado_anterior VARCHAR(20),
    estado_nuevo    VARCHAR(20),
    fecha_cambio    TIMESTAMP DEFAULT NOW(),
    usuario_db      TEXT      DEFAULT CURRENT_USER
);

-- 2. Función de auditoría
CREATE OR REPLACE FUNCTION fn_trg_auditoria_prestamos()
RETURNS TRIGGER AS $$
BEGIN
    -- Solo registrar si el estado cambió
    IF (OLD.estado <> NEW.estado) THEN
        INSERT INTO log_prestamos (prestamo_id, estado_anterior, estado_nuevo)
        VALUES (OLD.prestamo_id, OLD.estado::text, NEW.estado::text);
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 3. Trigger AFTER UPDATE sobre la tabla prestamos
CREATE TRIGGER trg_auditar_cambios
AFTER UPDATE ON prestamos
FOR EACH ROW
EXECUTE FUNCTION fn_trg_auditoria_prestamos();
🧪 Pruebas para la Clase
sql 
-- PRUEBA 1: Inserción → el ejemplar 1 debería cambiar estado automáticamente
INSERT INTO prestamos (ejemplar_id, usuario_id, fecha_limite)
VALUES (1, 1, '2026-06-01');

-- Verificar el efecto del trigger
SELECT * FROM ejemplares WHERE ejemplar_id = 1;

-- PRUEBA 2: Actualización → el trigger de auditoría guarda el cambio
UPDATE prestamos SET estado = 'Vencido' WHERE prestamo_id = 1;

-- Verificar que el log capturó el cambio
SELECT * FROM log_prestamos;

⚠️ Tips de Instructor SENA

Recursividad: Advierte sobre el peligro de crear un trigger que actualice la misma tabla que lo disparó (bucle infinito).
NEW y OLD: NEW contiene los datos entrantes (INSERT/UPDATE); OLD contiene los datos previos (UPDATE/DELETE).
Rendimiento: Muchos triggers pueden volver lenta una base de datos. Usarlos solo para lógica crítica de integridad.

-- == FUNCIÓN ==================================================
CREATE OR REPLACE FUNCTION fn_stock_disponible(p_isbn VARCHAR)
RETURNS INTEGER AS $$
DECLARE v_conteo INTEGER;
BEGIN
    SELECT COUNT(*) INTO v_conteo FROM ejemplares
    WHERE isbn = p_isbn AND estado IN ('Excelente','Bueno');
    RETURN v_conteo;
END;
$$ LANGUAGE plpgsql;

-- == PROCEDIMIENTO ============================================
CREATE OR REPLACE PROCEDURE sp_procesar_devolucion(
    p_prestamo_id INTEGER, p_nuevo_estado_fisico estado_ejemplar)
AS $$
BEGIN
    UPDATE prestamos SET estado = 'Devuelto'
    WHERE prestamo_id = p_prestamo_id;
    UPDATE ejemplares SET estado = p_nuevo_estado_fisico
    WHERE ejemplar_id = (SELECT ejemplar_id FROM prestamos
                         WHERE prestamo_id = p_prestamo_id);
    RAISE NOTICE 'Devolución procesada.';
END;
$$ LANGUAGE plpgsql;

-- == TRIGGERS =================================================
CREATE OR REPLACE FUNCTION fn_trg_gestionar_ejemplar()
RETURNS TRIGGER AS $$
BEGIN
    IF (TG_OP = 'INSERT') THEN
        UPDATE ejemplares SET estado = 'Deteriorado'
        WHERE ejemplar_id = NEW.ejemplar_id;
    ELSIF (TG_OP = 'DELETE') THEN
        UPDATE ejemplares SET estado = 'Bueno'
        WHERE ejemplar_id = OLD.ejemplar_id;
    END IF;
    RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_actualizar_ejemplar
AFTER INSERT OR DELETE ON prestamos
FOR EACH ROW EXECUTE FUNCTION fn_trg_gestionar_ejemplar();

CREATE TABLE log_prestamos (
    log_id SERIAL PRIMARY KEY, prestamo_id INTEGER,
    estado_anterior VARCHAR(20), estado_nuevo VARCHAR(20),
    fecha_cambio TIMESTAMP DEFAULT NOW(), usuario_db TEXT DEFAULT CURRENT_USER
);

CREATE OR REPLACE FUNCTION fn_trg_auditoria_prestamos()
RETURNS TRIGGER AS $$
BEGIN
    IF (OLD.estado <> NEW.estado) THEN
        INSERT INTO log_prestamos (prestamo_id, estado_anterior, estado_nuevo)
        VALUES (OLD.prestamo_id, OLD.estado::text, NEW.estado::text);
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
